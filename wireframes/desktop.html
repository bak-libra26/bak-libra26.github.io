JWT 만료 · 재발급 · 로그아웃 전략

spring-security jwt authentication
2026-03-10 · 8 min read

1. 왜 토큰 만료가 필요한가

JWT는 서버에 상태를 저장하지 않는 stateless 토큰입니다. 그래서 한 번 발급된 토큰은 서버에서 강제로 무효화할 수 없습니다. 만약 토큰이 탈취되면 만료될 때까지 악의적인 사용이 가능합니다.

이 문제를 완화하기 위해 Access Token의 만료 시간을 짧게 설정하고, 별도의 Refresh Token을 통해 재발급하는 전략을 사용합니다.

NOTE
Access Token의 일반적인 만료 시간은 15분~1시간, Refresh Token은 7일~30일 정도로 설정합니다.

1-1. Access Token vs Refresh Token

Access Token은 API 요청 시 인증에 사용되며, 짧은 수명을 가집니다. Refresh Token은 Access Token이 만료되었을 때 새로운 Access Token을 발급받는 데 사용됩니다.

JwtTokenProvider.java copy
public class JwtTokenProvider {
private static final long ACCESS_EXPIRY = 1000 * 60 * 30; // 30분
private static final long REFRESH_EXPIRY = 1000 * 60 * 60 * 24 * 14; // 14일
public String generateAccessToken(String username) {
return Jwts.builder()
.setSubject(username)
.setIssuedAt(new Date())
.setExpiration(new Date(System.currentTimeMillis() + ACCESS_EXPIRY))
.signWith(secretKey)
.compact();
}
}

2. 재발급 전략

클라이언트는 Access Token이 만료되면 Refresh Token을 서버에 보내 새로운 Access Token을 발급받습니다. 이 과정에서 Refresh Token의 유효성을 검증하고, 필요시 Refresh Token도 함께 갱신합니다.

Refresh Token Rotation은 Refresh Token이 사용될 때마다 새로운 Refresh Token을 발급하는 방식입니다. 이전 토큰은 즉시 무효화되어, 탈취된 토큰의 재사용을 방지할 수 있습니다.

2-1. Refresh Token Rotation

Refresh Token Rotation에서는 서버가 refreshToken 요청을 받으면, 새 Access Token과 함께 새 Refresh Token도 발급합니다. 이전의 Refresh Token은 블랙리스트에 등록하거나 DB에서 삭제합니다.

3. 로그아웃 처리

JWT 기반 인증에서 로그아웃은 서버가 토큰을 무효화해야 하므로, 순수한 stateless 방식만으로는 불가능합니다. 대표적인 전략은 Redis에 토큰 블랙리스트를 관리하거나, Refresh Token을 DB에서 삭제하는 것입니다.

클라이언트 측에서도 토큰을 삭제하고, 서버 측에서는 /api/auth/logout 엔드포인트에서 Refresh Token을 무효화합니다.
